# MatrixOne Git4Data 技术详解（八）·AI 训练实践篇：机器学习的持续学习——增量还是重训，先看清"数据动了哪些"

从这一篇起，进入 **AI 训练** 主题。先从一个所有机器学习工程师都熟的循环说起：

> 数据每天都在变——新样本进来、旧标签被修正。于是每周（甚至每天）把**全量**数据重新喂给模型，从头训一遍。数据涨到千万级后，这个循环越来越贵、越来越慢——但你不敢省，因为你**说不清这周到底哪些数据变了**。

先说清楚一件容易被带偏的事：**这一篇不卖"增量训练一定比全量好"**。增量训练有它适合和不适合的场景（下面掰开讲）。真正普适、却又常常缺失的，是更上游的一个答案——**"上次训练之后，数据到底动了哪些？"** 有了它，你才谈得上"该增量就增量、该重训就重训"，也才谈得上复现、归因、回退。而把这个答案变成一条 SQL，正是 MatrixOne 的 git4data 能力最擅长的。SQL 全部在 MatrixOne `4.0.0-rc3` 上实测过。

> 📦 本文 SQL 整体可跑：[matrixorigin/git4data-tutorial](https://github.com/matrixorigin/git4data-tutorial) 的 `08-ml-incremental/`。环境：`docker run -d -p 6001:6001 --name matrixone matrixorigin/matrixone:4.0.0-rc3`。

## 先泼盆冷水：增量训练到底适合什么？什么时候必须全量重训？

"数据变了就只训变的那部分"——听着高效，但**增量训练是一个有明确适用边界的训练技术，不是万能开关**。先把账算清楚。

**增量训练（online / `partial_fit` / warm-start）适合：**

- **模型本身支持增量更新**：SGD 系线性模型、朴素贝叶斯、部分神经网络的继续训练（continued training / 微调）；
- **数据分布相对稳定**：新数据是同一分布的延伸，增量更新能平稳收敛；
- **数据大、重训又频繁**：全量重训贵到不可接受，而每轮变化只占极小比例；
- **只是"多了一批新样本"**：纯追加，没有对旧数据的大量订正或删除。

**但很多时候，你就是需要全量重训——硬上增量反而会坑你：**

- **灾难性遗忘**：只拿新数据更新，神经网络容易**忘掉旧知识**、在老分布上掉点。纯"只训增量"可能让模型整体变差。
- **分布漂移（concept drift）**：分布真变了，正确做法往往是在**全量（或重采样、对旧数据降权）**上重训，而不是简单追加新数据。
- **删除 / 订正了旧数据**：最容易被忽略的一条。**从数据集删一行，并不能把它对已训模型的影响删掉**——这就是"机器遗忘（machine unlearning）"难题。旧标签被证明错了、或某些样本要合规删除时，通常**只有重训**才能真正抹掉它的影响，增量做不到。
- **非增量算法**：随机森林、XGBoost 这类树模型大多不支持增量更新，本来就得重训。
- **改了特征 / 超参 / 架构**：任何这类改动都要求全量重训。
- **要一个干净、可复现、可审计的模型**：路径依赖的增量模型，有时不如一次干净的全量重训。

一句话：**增量 vs 全量，是个因场景而定的训练决策，没有"永远更好"的一方。** 所以这一篇真正的主角，不是"逼你做增量"，而是那个**两种选择都用得上的前提**——"数据动了哪些"。

## 无论增量还是全量，你都要先回答"数据动了哪些"

想清楚上面的取舍，你会发现两条路卡在**同一个前提**上：

- **要做增量**？那你必须精确拿到"这轮新增 / 改动的行"，才能只把这些喂给 `partial_fit`；
- **要做全量重训**？那你得先知道"变了多少、变了什么"，才能**判断到底该不该重训**（小追加 → 增量够了；大漂移、大量订正 / 删除 → 触发全量），而且重训时还需要一个**确切、可复现**的训练集版本。

这个前提——"相对上次训练，数据动了哪些"——恰恰是大多数团队最缺、也最难可靠拿到的。下面这套三步循环，就是把它变成一条 SQL。

---

## 三步循环：pin → 训练 → DIFF 出变化

训练集是一张 `samples` 表，旁边放一张模型注册表 `model_registry`。

### 第一步：训练前，pin 一个版本

```sql
CREATE SNAPSHOT train_v1 FOR TABLE mltrain_demo samples;   -- 毫秒级、近零成本
-- （训练器读全表，训出模型 m1 …）
INSERT INTO model_registry VALUES ('m1', 'train_v1', 0.9012);
```

这一步毫秒级、几乎不占空间（第三篇讲过原理），但它把"模型 m1 是用什么数据训的"从一句口头描述，变成了一个**可执行的事实**——任何时候 `SELECT … FROM samples {SNAPSHOT='train_v1'}` 都能逐位复原 m1 的训练集。

### 第二步：一周后，数据动了——但动了哪些？

一周的真实生活：新批次进来 3000 行，质检又修正了 200 个标签：

```sql
INSERT INTO samples SELECT 100000 + result, … FROM generate_series(1, 3000) g;  -- 新数据
UPDATE samples SET label = 1 - label WHERE sample_id BETWEEN 500 AND 699;       -- 标签修正
```

现在问那个关键问题——**相对 m1 的训练集，数据到底变了哪些？** 一条 DIFF：

```sql
DATA BRANCH DIFF samples AGAINST samples {SNAPSHOT='train_v1'} OUTPUT SUMMARY;
--   INSERTED = 3000   (新批次)
--   UPDATED  =  200   (被修正的标签)
--   DELETED  =    0
```

答案精确到行：**变更就是这 3200 行，其余 10 万行一行没动。** 拿到这个答案，接下来是**你的训练决策**：

- 若场景适合增量（纯追加、分布稳定）→ 把这批变化取出来喂 `partial_fit`：

  ```sql
  -- 增量训练集 = 相对 train_v1「新增或改过」的行（按值比对，净变化）
  SELECT * FROM samples cur
  WHERE NOT EXISTS (
      SELECT 1 FROM samples {SNAPSHOT='train_v1'} base
      WHERE base.sample_id = cur.sample_id
        AND base.f1 = cur.f1 AND base.f2 = cur.f2 AND base.label = cur.label
  );
  --   实测返回 3200 行（全量重训则要处理整张 103000 行的表）。
  ```

- 若这 200 处修正其实是**大规模标签订正**、或数据发生了**漂移** → 那就别硬增量，在新版本上**全量重训**。而 DIFF 给的"变了多少、变了什么"，正是你做这个判断的依据。

无论走哪条，DIFF 都先把决策建立在事实上，而不是"感觉这周变化不大"。

### 第三步：训练完，再 pin 一个

```sql
CREATE SNAPSHOT train_v2 FOR TABLE mltrain_demo samples;
INSERT INTO model_registry VALUES ('m2', 'train_v2', 0.9145);
```

于是注册表里积累出一条**模型 ↔ 数据的对应链**（这条链的价值，和你选增量还是全量**无关**——都成立）：

```
m1 ← train_v1 (100,000 行)
m2 ← train_v2 (103,000 行) = train_v1 + 3000 新增 + 200 修正
```

它解锁了几个平时做不到的动作：

- **精确复现**：三个月后审计问"m1 是用什么训的"，`SELECT … {SNAPSHOT='train_v1'}` 一查便知，逐位一致（实测 10 万行原样复原）；
- **归因调试**：m2 比 m1 差了？把两个快照 DIFF 一下，可疑变更就是那 3200 行，而不是在 10 万行里大海捞针；
- **数据回退**：发现修正的标签本身是错的，`RESTORE TABLE … {SNAPSHOT = train_v1}` 回到上一版，重新来过。

## 当增量确实适用时，这笔账省多少？

**前提是"增量适用"**（纯追加、分布稳定、模型支持）。在这个前提下，我们在配套实验里量化过：同一场景跑 **6 轮**，每轮新增约 1000 行、偶有几处标签订正——

- **全量重训**：每轮都要处理**当前整张表**，6 轮累计处理 **21,000** 行；
- **只训增量**：每轮只处理该轮的变化，6 轮累计只处理 **6,004** 行。

关键不在这一次的 3.5 倍，而在**趋势**：全量重训的累计成本随轮数**二次增长**，增量近似**线性**。实测里每一轮的增量都稳定在"约 1000 行"，哪怕那时整张表已是它的 6 倍。

> 但请记住上面那盆冷水：**如果某一轮发生了漂移、或有大量旧数据被删改而必须重训，这条"省算力"就不成立**——那一轮你就该老实全量重训。此时快照 + DIFF 仍然给你复现、归因、回退，只是不省这份算力。省不省，取决于场景；能不能说清"变了哪些"、能不能复现，才是普适的。

## 别的方案怎么拿到"变了哪些"？问题在哪？

不管你最终增量还是全量，"相对上次训练数据动了哪些"这个答案都得先拿到。看看常见做法——每一种都能凑合，但都在某个地方卡住。

**方案 A：每次全量重训、不追踪变化（基线）。** 干脆不问"变了啥"，每轮 O(N) 从头再来。数据小无所谓，大了就是纯烧钱，而且你依旧没有复现 / 归因的能力。

**方案 B：`updated_at` 水位线。** 加个 `updated_at`，记住"上次训练到哪个时刻"，下轮 `WHERE updated_at > 水位线`。坑不少：**漏删除**（DELETE 掉的行查不出来）；**依赖全链路纪律**——任何不更新 `updated_at` 的批量回填 / 订正都会被漏掉；**水位线是移动指针，不是版本**——你没法"复现上次训练的确切集合"，也没法 diff 任意两个历史版本；一行改了又改回，它照样算你头上。

**方案 C：CDC / binlog 流（Debezium + Kafka）。** 把每行变更实时流出去、下游消费。问题：**基建重**（Kafka + Debezium + 消费者一整套）；拿到的是**变更事件洪流**，不是"相对某个训练版本的净增量"；要对齐"m1 是哪个版本训的"得回放 offset；精确一次很折腾；一行改 5 次给你 5 条事件。

**方案 D：自己存两份全量 + `EXCEPT` / 反连接。** 留一份上次训练的全量副本做差集。问题：**每个训练版本一份全量副本 → N 倍存储**；算差集是**两份全量的全表扫**（O(N)——为了找那点增量又碰了所有数据）；版本一多就撑不住。

**方案 E：数据版本化工具。**
- **DVC**：版本化**文件**——任何改动都产生新文件、重算 hash；能比文件版本，但**不是行级**"哪些行变了"，粒度是整个文件。
- **lakeFS**：版本化对象存储里的文件 / 路径，diff 是**对象 / 文件级**，不是行级。
- **Delta Lake（time travel + Change Data Feed）**：这个**真能**给版本间的**行级**变更，是最接近的一个。差异：要**显式开启 CDF**（额外写 change 文件）、经 Spark 消费、面向湖 / 分析；给的是**变更事件**（可能含中间态），而非"相对某快照的净 diff"；且不是一个还在对外服务点查的库。

| 方案 | 捕获 增/改/删 | 相对指定版本的行级净增量 | 算增量的成本 | 额外基建 / 存储 | 能逐位复现训练集 | 能 diff 任意两版 |
|---|---|---|---|---|---|---|
| A 全量重训 | —（不区分） | — | 每轮 O(N) | 无 | 否 | 否 |
| B updated_at 水位 | 增 / 改（**漏删**） | 否（只有"之后改过"） | O(增量) | 要维护列 + 纪律 | 否 | 否 |
| C CDC / binlog 流 | 增 / 改 / 删 | 否（事件流，非净差） | 流式 | **重（Kafka+CDC）** | 难 | 难 |
| D 双副本 + EXCEPT | 增 / 改 / 删 | 是 | **O(N) 全表扫** | **N 倍存储** | 是（存了副本） | 副本在才行 |
| E1 DVC / lakeFS | 文件 / 对象级 | 否（非行级） | 文件级 | 另一套工具 | 是（文件版本） | 文件级 |
| E2 Delta CDF | 增 / 改 / 删 | 事件（可能含中间态） | Spark 消费 | 开 CDF + Spark | 是（time travel） | 版本级 |
| **MatrixOne（git4data 能力）** | **增 / 改 / 删** | **是，一条 `DATA BRANCH DIFF … {SNAPSHOT}`** | **只随变更量** | **无（就是 SQL）** | **是（快照逐位复现）** | **是（任意两快照）** |

一句话：别的方案要么**只能近似**（水位线漏删、DVC 非行级），要么**代价大**（双副本 N 倍存储、CDC 一套流），要么**在湖上、要另配引擎**（Delta CDF）。MatrixOne 把它收成一条 SQL——相对任意 pin 过的版本，`DATA BRANCH DIFF` 给出**行级净增量**，成本只随变更量、不随表大小；而且同一个快照既是"增量的基准"，也是"可逐位复现的训练集版本"。这就是 git4data 这套能力落在一个 HTAP 数据库里的好处：**版本、增量、复现，是同一件事的三个面。**

## 成本与边界

- **快照毫秒级、与数据量无关**（第三篇原理）；**DIFF 只随变更量走**，不扫全表。
- **增量训练的坑，数据库不替你兜**：灾难性遗忘、漂移、机器遗忘，都是**训练侧**的问题——git4data 给的是"精确的变化 + 可复现的版本"，而**要不要增量、怎么训，是你的判断**。别把"能拿到增量"误当成"就该做增量"。
- **DIFF 报的是"自快照以来被动过的行"**（按是否改过算，不做值比对）：拿来喂增量训练正合适；若你要的是"当前值和上一版不同"的**净变化**，用上面那条**按值反连接**（它会自然排除"改了又改回"的行）。
- **要能复现，就别急着删快照**：被快照钉住的历史版本会占存储，直到 `DROP SNAPSHOT`。给每个上线模型的 `train_vN` 长期保留、给废弃中间版本设清理策略。
- **行级 DIFF 要求 schema 一致**（第四篇边界）：训练集要加特征列，先在主线改 schema、再继续。

## 结语

把这一篇收成一句话：**先精确知道"数据动了哪些"，再决定增量还是重训。** 三步循环——

```
①  CREATE SNAPSHOT train_vN            -- 训练前，钉住数据版本
②  训练 → 注册 (model, train_vN)        -- 模型与数据版本绑定
③  下一轮：DIFF 现状 AGAINST train_vN   -- 变化 = 确切的行 → 据此决定增量 / 全量
```

它省的不一定是算力（那取决于场景适不适合增量），但它一定给你三样平时**根本拿不到**的东西：模型的**可复现**、退步的**可归因**、脏数据的**可回退**。

下一篇换到 LLM 的语境：**SFT 数据策展**——几十万条指令数据的去重、过滤、去污染，全用 SQL 原地完成，而且每一刀都有 DIFF 作为"收据"，删了什么、为什么删、能不能撤，都说得清。
